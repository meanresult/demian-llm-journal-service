----------------------------------------------------
1. 보안 (Security)
----------------------------------------------------

[오늘 발견된 문제]
- .gitignore에 .env가 있었지만 공백 오타로 실제로는 무효였음
- get_token.py에 이메일/패스워드 하드코딩
- 배포 후 Railway에 DATABASE_URL 넣을 때 줄바꿈 공백이 섞여 들어감

[신입이 반드시 알아야 할 것]
- .env 파일은 git에 절대 들어가면 안 된다.
  → git log --all --full-history -- .env 로 항상 확인
  → 이미 커밋됐다면 키 즉시 rotate (파일 삭제만으로는 히스토리에 남음)

- 코드에 비밀값(API key, 패스워드) 직접 쓰지 말 것
  → 테스트용이라도 os.getenv() 또는 .env 파일 사용

- 환경변수 복사할 때 줄바꿈/공백 섞이는 경우 많음
  → 배포 후 첫 번째로 확인할 것


----------------------------------------------------
2. DB 설계 및 마이그레이션
----------------------------------------------------

[오늘 겪은 문제]
- init_db.py의 create_all()은 기존 테이블을 수정하지 않음
- 새 컬럼(reflection, structured_data) 추가 후 서버가 500 에러
- 해결책: 테이블 DROP 후 재생성 (데이터 없었으니 가능했지만 운영중이었다면 데이터 손실)

[신입이 반드시 알아야 할 것]
- create_all()은 개발 초기에만 쓸 수 있는 방법이다.
  데이터가 생기는 순간 Alembic 같은 마이그레이션 툴 필요.

- Alembic 기본 흐름:
  alembic init alembic
  alembic revision --autogenerate -m "add reflection column"
  alembic upgrade head

- DB 스키마 변경은 배포 전에 항상 마이그레이션 파일 먼저 준비할 것.
  코드 배포와 DB 변경 순서가 잘못되면 다운타임 발생.

- timezone 처리: DateTime(timezone=True) + server_default=func.now() 가 맞다.
  datetime.utcnow() 직접 쓰는 건 deprecated 패턴.


----------------------------------------------------
3. LLM 파이프라인 설계
----------------------------------------------------

[오늘 선택한 방법]
- FastAPI BackgroundTasks로 비동기 처리
- 저장 즉시 응답 → LLM은 뒤에서 처리 → reflection=null이면 클라이언트가 "생성 중" 표시

[신입이 알아야 할 트레이드오프]
- BackgroundTasks는 간단하지만 서버가 재시작되면 처리 중인 작업이 사라짐
  → 유저 수가 늘면 Celery + Redis 또는 AWS SQS 같은 큐 시스템 고려
  → 지금 규모에서는 BackgroundTasks로 충분

- LLM 호출은 반드시 retry + exponential backoff 적용
  → 네트워크 일시 오류, Rate Limit 등으로 실패할 수 있음
  → 오늘 구현: 최대 3회, 1초→2초→4초 대기

- LLM 비용 관리:
  → OpenAI gpt-4o-mini: 저렴하고 구조화에 적합
  → Claude Haiku: 리플렉션 생성에 적합하나 크레딧 필요
  → 프롬프트가 5000 토큰 넘어가면 비용 경고 필요

- Pydantic으로 LLM 응답 검증 필수
  → LLM은 가끔 형식 안 맞는 JSON 반환함
  → model_validate_json()으로 바로 잡을 것

4. API 설계
----------------------------------------------------

[오늘 한 것들]
- GET /diary-entries: 인증 없이도 다른 유저 데이터 조회 가능했던 stub 코드 제거
- /users/me: 첫 호출 시 users 테이블 자동 생성 (upsert)

[신입이 알아야 할 것]
- 인증(Authentication) vs 인가(Authorization) 구분:
  → 인증: 이 사람이 누구인가 (JWT 검증)
  → 인가: 이 사람이 이 데이터에 접근할 권한이 있나 (WHERE user_id = :user_id)
  → 오늘 코드에서 diary_entries는 항상 user_id로 필터링 → 인가 처리됨

- Pydantic 스키마(request/response)를 먼저 정의하고 구현하면 실수 줄어듦
  → 스키마 없으면 어떤 데이터가 오고 가는지 불명확

- HTTP 상태코드 구분:
  → 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Server Error
  → 오늘 POST /diary-entries에 status_code=201 명시적으로 붙인 이유

5. 배포 및 인프라
----------------------------------------------------

[오늘 겪은 문제]
- Dockerfile에 PORT 하드코딩 → Railway 502 에러
- DATABASE_URL에 공백 섞임 → 서버 크래시

[신입이 알아야 할 것]
- 클라우드 플랫폼(Railway, Render, Fly.io)은 PORT 환경변수를 동적으로 할당
  → CMD에 ${PORT:-8000} 패턴 필수

- 배포 후 체크리스트:
  1. /health 엔드포인트 응답 확인
  2. 환경변수 누락 없는지 확인
  3. DB 연결 확인 (/test-db)
  4. 인증 필요한 API 실제 토큰으로 확인

- Supabase Free 티어: 7일 비활성 시 자동 pause
  → cron-job.org 등으로 주기적 핑 필요 (배포 후 설정할 것)

- 로컬 개발 vs 배포 환경 차이:
  → 로컬: .env 파일
  → Railway: Variables 탭
  → Vercel: Environment Variables 설정
  → 세 곳 모두 동기화되어 있는지 확인

6. 코드 품질
----------------------------------------------------

[오늘 정리한 것들]
- Session.py → session.py (파일명 소문자 통일)
- print() → logging 교체
- bare except → 구체적인 예외 타입 명시 (JWTError 등)
- UNUSE_database.py 같은 사용하지 않는 파일은 삭제

[신입이 습관화해야 할 것]
- print()는 개발 중에만, 운영 코드에는 반드시 logging 사용
  → logging.INFO: 정상 흐름
  → logging.WARNING: 예상 가능한 오류 (JWT 만료 등)
  → logging.ERROR: 처리 실패 (LLM 호출 실패 등)

- 함수/파일명은 역할을 명확히:
  → get_token.py처럼 테스트용 파일은 scripts/ 폴더로 분리
  → 실제 서비스 코드와 섞이지 않게

- 커밋 메세지 컨벤션 (Conventional Commits):
  feat: 새 기능
  fix: 버그 수정
  refactor: 동작 변경 없는 코드 개선
  chore: 설정, 빌드 관련
  docs: 문서

7. 아키텍처 관점
----------------------------------------------------

[현재 구조]
frontend/ (Next.js) → backend/ (FastAPI) → Supabase DB
                                         → OpenAI API
                                         → Claude API (크레딧 충전 후)

[지금 당장은 괜찮지만 유저 늘면 고려할 것]
- DB 연결 풀 설정: 현재 기본값 → pool_size, max_overflow 명시 필요
- JWKS 캐싱: 오늘 1시간 캐싱 구현 → 운영에서 충분
- BackgroundTasks → 유저 많아지면 Celery + Redis로 교체
- rate limiting: 악성 사용자가 API 남용 방지 (slowapi 라이브러리)
- CORS: 현재 allow_origins=["*"] → 배포 후 프론트 도메인으로 교체 필수


----------------------------------------------------
8. 오늘 질문에서 배운 핵심
----------------------------------------------------

Q. Supabase 말고 실무에서 뭘 써?
A. 유저 적으면 Supabase가 가장 가성비 좋음.
   SQLAlchemy 쓰면 나중에 DB 교체 비용 낮음.

Q. 돈을 쓰고 싶지 않다면?
A. Supabase Free + cron-job.org 핑 조합.
   진짜 무료로 오래 쓸 수 있음.

Q. LLM 파이프라인 동기 vs 비동기?
A. 실무에서는 대부분 비동기.
   LLM 실패해도 데이터 저장은 보장되어야 함.

Q. 프론트 별도 레포 vs 같은 레포?
A. 혼자 개발, 소규모면 모노레포가 관리 편함.
   팀 커지면 분리.

Q. Docker에서 PORT 고정하면 왜 문제?
A. 클라우드 플랫폼은 포트를 동적 할당함.
   앱이 플랫폼이 준 포트로 들어야 함.
